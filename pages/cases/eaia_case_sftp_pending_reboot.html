Un reboot Windows
qui aurait pu couper
la production usine.

Une alerte Windows Update pending reboot sur SFTP-01 semblait routinière. L’agent a plutôt identifié un rôle SFTP automatisé, observé les cycles de transfert, placé le redémarrage entre deux échanges et exigé une validation fonctionnelle réelle avant fermeture.

47,5jours d’uptime
30minutes par cycle
0session active
2reboots supervisés
1transfert validé
Pourquoi ce cas
vaut une page de démonstration.

La valeur n’est pas le redémarrage. La valeur est le jugement : reconnaître qu’un serveur en apparence simple porte un flux métier automatisé et qu’une maintenance mal placée peut interrompre une chaîne de production.

Signal initial

Pending reboot réel

CBS, Windows Update et PendingFileRename confirmaient qu’un redémarrage était requis pour finaliser la maintenance Windows.

Risque caché

Service SFTP actif

Les journaux OpenSSH indiquaient des sessions internal-sftp courtes et récurrentes, avec un cycle observé aux 30 minutes.

Décision

Fenêtre entre cycles

Le reboot a été exécuté seulement après fermeture normale d’un transfert et validation de l’absence de connexion active.

Pipeline
opérationnel.
Précheck
  • Uptime et flags pending reboot
  • Services OpenSSH
  • Journaux SFTP
  • Mode maintenance RMM
Intervention
  • Attente du prochain cycle
  • Validation de l’absence de session
  • Redémarrage supervisé
  • Second reboot pour finalisation Windows
Postcheck
  • sshd et ssh-agent Running
  • Port SFTP en écoute
  • Pending reboot revenu à False
  • Transfert automatisé post-reboot validé
